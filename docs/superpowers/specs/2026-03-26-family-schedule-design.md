# 「共享日程」产品设计文档

> 版本：v1.0 | 日期：2026-03-26
> 产品定位：为共享而生的智能日程 — 以"人与人的日程连接"为核心

---

## 一、市场背景与机会

### 1.1 市场规模

| 指标 | 数据 | 来源 |
|------|------|------|
| 全球共享日历APP市场（2024） | 3.04亿美元 | QYResearch |
| 全球共享日历APP市场（2031预测） | 6.71亿美元 | QYResearch |
| CAGR | 12.0% | QYResearch |
| 亚太市场份额 | 16%（增长潜力最大） | QYResearch |
| 中国日历应用市场CAGR | 12.3% | QYResearch |

### 1.2 竞争格局

| 细分领域 | 饱和度 | 代表产品 |
|---------|--------|---------|
| 传统万年历 | 高度饱和 | 中华万年历（3亿用户，广告泛滥） |
| 企业协同日程 | 较饱和 | 飞书、钉钉、企业微信 |
| 个人效率/待办 | 高度饱和 | 滴答清单、微软待办 |
| **家庭/社交共享日程** | **严重不饱和** | **无强势本土玩家** |
| AI智能日程 | 空白 | 中国无对标产品 |

### 1.3 核心竞品分析

| 竞品 | 定位 | 共享能力 | 中国市场局限 |
|------|------|---------|-------------|
| TimeTree（日本） | 家庭共享日历，7000万用户 | 核心功能 | 未本土化，无微信集成，无农历 |
| 腾讯日历 | 办公+生活日历 | 有但偏弱 | 共享功能基础，偏办公场景 |
| 飞书/钉钉日历 | 企业协同 | 企业内强大 | 仅面向B端，家庭场景不适用 |
| Google Calendar | 全能日历 | 极强 | 需翻墙，基本不可用 |
| Apple Calendar | 苹果生态日历 | iCloud共享 | 仅苹果设备，排除安卓用户 |
| 朝夕日历（已停） | 语音+社交日历 | 有 | 被字节收购后停摆，但验证了方向 |

### 1.4 用户需求调研（多平台综合）

**知乎**（讨论最深入）：
- 跨平台同步是第一痛点："一方iPhone一方安卓，怎么同步？"
- 高频评价："这个领域竟然没有一个让人耳熟能详的产品"
- TimeTree被多次推荐但用户期望有本土化替代

**小红书**：
- "情侣必备app"合集中常提及共享日历需求
- 用户偏好高颜值、无广告、简洁的工具
- 吐槽国内缺少真正好用的共享日历

**抖音**：
- 共享日历品牌声量极低，说明市场认知空白
- "异地恋必备工具"视频中偶有提及日程同步需求
- 用户更接受免费+内购装饰的模式

**B站**：
- 效率工具UP主偶有提及，无专门测评
- 学生和年轻群体有"共享日程"刚需

**微博/豆瓣/贴吧**：
- 讨论极少，品类尚未被市场教育

### 1.5 核心机会

1. **家庭共享日程的本土化空白** — TimeTree验证需求（7000万用户/60%家庭），但中国无本土强者
2. **AI+语音日程在中国完全空白** — Motion/Reclaim模式无中国对标
3. **微信生态 = 天然分发渠道** — 9.49亿小程序用户，89.2%渗透率
4. **朝夕日历停摆留下的真空** — 已验证"语音+微信+日历"方向

## 二、产品定位

### 2.1 一句话定位

**"为共享而生的智能日程"** — 不是个人日历附带共享，而是以人与人的日程连接为核心。

### 2.2 目标用户

通用共享场景，不限关系类型：

| 场景 | 典型用户 | 核心需求 |
|------|---------|---------|
| 家庭 | 双职工父母、祖孙三代 | 接送协调、家庭活动、老人关怀提醒 |
| 情侣 | 恋爱中/异地恋 | 约会安排、纪念日、空闲时间同步 |
| 朋友 | 闺蜜团、兄弟群 | 聚会组织、旅行计划 |
| 小团队 | 兴趣小组、志愿者 | 活动排期、值班安排 |

### 2.3 差异化三角

```
          共享优先
         /        \
        /   本产品  \
       /            \
  AI语音驱动 ———— 微信生态深度绑定
```

1. **共享优先** — 首屏即共享视图，日历组是一等公民
2. **AI语音驱动** — 说一句话创建日程，LLM语义解析
3. **微信原生体验** — 社交关系链 + 分享裂变 + 消息提醒

## 三、产品形态与技术策略

### 3.1 首发形态

**微信小程序**

| 优势 | 说明 |
|------|------|
| 低获客成本 | 微信社交裂变，邀请即增长 |
| 快速验证 | 开发周期短，快速迭代 |
| 零安装门槛 | 扫码/链接即用，降低共享阻力 |
| 社交传播 | 分享卡片在微信聊天中原生展示 |

| 限制 | 应对策略 |
|------|---------|
| 无后台推送 | 使用微信订阅消息 + 服务通知 |
| 无主屏Widget | 引导用户"添加到桌面" |
| 功能受限 | 核心功能在小程序实现，深度功能留给后续App |

### 3.2 演进路线

| 阶段 | 形态 | 目标 |
|------|------|------|
| Phase 1（MVP） | 微信小程序 | 验证PMF，跑通核心共享流程 |
| Phase 2 | 小程序优化 + 独立App开发 | 根据数据决定App优先级 |
| Phase 3 | 小程序（轻量入口）+ App（完整体验） | 双端协同 |
| Phase 4 | 英文版 | 视中国市场反馈决定 |

### 4.2 模块详细设计

---

#### 模块1：日程管理

**1.1 手动创建日程**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 标题 | 文本 | 是 | 日程名称 |
| 日期 | 日期选择器 | 是 | 支持公历/农历切换 |
| 时间 | 时间选择器 | 否 | 不填则为全天事件 |
| 持续时长 | 选择器 | 否 | 15min/30min/1h/2h/自定义 |
| 重复 | 选择器 | 否 | 不重复/每天/每周/每月/每年/自定义 |
| 所属日历 | 选择器 | 是 | 个人日历 或 某个共享日历组 |
| 可见性 | 选择器 | 是 | 公开/仅组内可见/私密 |
| 提醒 | 多选 | 否 | 不提醒/提前5min/15min/1h/1天/自定义 |
| 地点 | 文本 | 否 | 支持输入地址 |
| 备注 | 富文本 | 否 | 补充说明 |
| 颜色标签 | 颜色选择 | 否 | 快速区分不同类型 |

**1.2 语音创建日程（AI）**

**用户流程：**
```
用户点击语音按钮 → 说话 → 语音转文字（微信API/第三方ASR）
→ 文字发送给LLM → LLM提取结构化日程信息 → 预填表单 → 用户确认/修改 → 创建
```

**示例：**
| 用户说 | AI解析结果 |
|--------|-----------|
| "下周三下午三点接小明放学" | 标题：接小明放学，日期：下周三，时间：15:00 |
| "每周五晚上7点和老婆一起健身" | 标题：和老婆一起健身，日期：每周五，时间：19:00，重复：每周 |
| "后天上午提醒妈妈去医院体检" | 标题：妈妈体检，日期：后天，时间：上午（默认9:00），提醒：是 |
| "国庆节家庭聚餐" | 标题：家庭聚餐，日期：10月1日，全天事件 |

**技术方案（MVP）：**
- 语音转文字：微信小程序 `wx.getRecorderManager()` + 微信同声传译插件 或 腾讯云ASR
- 语义解析：调用国产LLM API（DeepSeek/通义千问/文心一言），Prompt工程提取结构化信息
- 输出格式：JSON `{ title, date, time, duration, repeat, reminder, location }`
- 用户始终有确认/修改步骤，AI不会直接创建日程

**1.3 日历视图**

| 视图 | 说明 |
|------|------|
| 日视图 | 当日时间轴，显示所有日程，不同日历组用不同颜色区分 |
| 周视图 | 7天网格，快速查看一周安排 |
| 月视图（默认） | 月历格式，日期格子中显示日程点/缩略，农历日期同步显示 |
| 列表视图 | 按时间线排列的日程列表，适合查看近期所有安排 |

**1.4 农历/节假日/调休**

- 每日同时显示公历和农历日期
- 中国法定节假日标注（春节、国庆、中秋等）
- 调休工作日标注（"班"字标识）
- 二十四节气显示
- 数据来源：国务院每年发布的节假日安排 + 农历算法库

**1.5 日程模板**

预设常用场景模板，用户可一键使用：

| 模板名 | 预设内容 |
|--------|---------|
| 每周接娃 | 标题：接[孩子名]放学，重复：每周一至五，时间：待填 |
| 家庭聚餐 | 标题：家庭聚餐，提醒：提前1天 |
| 纪念日 | 标题：[纪念日名]，重复：每年，提醒：提前3天 |
| 定期体检 | 标题：[谁]体检，提醒：提前1周 |
| 周末活动 | 标题：待填，日期：本周六/日 |

用户也可将自己的日程"保存为模板"。

#### 模块2：共享日历组

**2.1 日历组概念**

日历组是产品的核心概念。每个组是一个独立的共享日历空间：

```
用户 ─┬─ 个人日历（默认，私有）
      ├─ 日历组"我们家"（家庭成员共享）
      ├─ 日历组"和小李"（情侣共享）
      └─ 日历组"周末篮球"（朋友共享）
```

**2.2 创建日历组**

| 字段 | 说明 |
|------|------|
| 组名 | 如"我们家"、"闺蜜团" |
| 图标/封面 | 预设图标 或 自定义上传 |
| 颜色 | 日历组的标识色，在视图中区分 |
| 简介 | 可选，描述组的用途 |

**2.3 邀请成员**

三种邀请方式：
1. **微信好友邀请** — 生成小程序分享卡片，好友点击即加入
2. **链接邀请** — 生成邀请链接，可复制到任何聊天
3. **二维码邀请** — 生成二维码，适合面对面邀请

**2.4 成员管理**

| 角色 | 权限 |
|------|------|
| 创建者 | 全部权限，包括解散日历组 |
| 管理员 | 管理成员、编辑所有日程 |
| 成员 | 查看组内日程、创建/编辑自己的日程 |

MVP限制：
- 免费版：最多创建 3 个日历组，每组最多 10 人
- 付费版：无限日历组，每组最多 50 人

#### 模块5：通知提醒

**5.1 微信消息通知**

利用微信小程序消息能力：

| 通知类型 | 触发条件 | 微信消息能力 |
|---------|---------|-------------|
| 日程提醒 | 用户设置的提醒时间到达 | 订阅消息（需用户授权） |
| 日程变更 | 组内成员修改/删除了共享日程 | 订阅消息 |
| 新日程通知 | 组内有人创建了新日程 | 订阅消息 |
| 邀请通知 | 被邀请加入日历组 | 服务通知 |

**5.2 订阅消息策略**

微信订阅消息限制：每次用户操作只能获取一次发送权限。

应对策略：
- 在关键操作节点引导用户授权（如创建日程后："是否接收提醒通知？"）
- 提供"长期订阅"选项（微信支持的类目下）
- 设计合理的授权引导，避免频繁打扰

**5.3 提醒时间选项**

| 提前1天 | |
| 提前3天 | 适合重要事项 |

#### 模块6：空闲时间匹配

**6.1 功能流程**

```
选择日历组/成员 → 选择日期范围 → 系统分析成员日程
→ 高亮显示所有人都空闲的时间段 → 用户选择时段 → 一键创建日程
```

**6.2 界面设计**

```
┌──────────────────────────────────┐
│ 找空闲时间                  本周 ▼ │
│                                    │
│ 参与人：小明妈妈 ✓ 小明爸爸 ✓ 奶奶 ✓│
│                                    │
│ 周一  ░░░░████░░░░░░████░░░░      │
│ 周二  ░░████░░░░░░░░░░░░████      │
│ 周三  ░░░░░░░░████░░░░░░░░░░      │
│ 周四  ░░████████░░░░░░░░░░░░      │
│ 周五  ░░░░░░░░░░████░░░░████      │
│ 周六  ░░░░░░░░░░░░░░░░░░░░░░      │
│ 周日  ░░░░░░░░░░░░░░░░░░░░░░      │
│                                    │
│ █ = 有人忙碌  ░ = 全员空闲          │
│                                    │
│ 建议时段：                          │
│ · 周三 上午 9:00-12:00 [创建日程]   │
│ · 周六 全天           [创建日程]    │
│ · 周日 全天           [创建日程]    │
└──────────────────────────────────┘
```

**6.3 算法逻辑**

1. 获取所选成员在日期范围内的所有日程
2. 排除"私密"日程的具体信息，但保留时间段占用
3. 计算所有成员的空闲时间交集
4. 按时长排序，优先推荐较长的连续空闲段
5. 过滤掉不合理时段（如凌晨）


## 六、数据模型

### 6.1 核心实体

```
User（用户）
├── id: string (UUID)
├── openid: string (微信OpenID)
├── unionid: string (微信UnionID，可选)
├── nickname: string
├── avatar: string (URL)
├── settings: JSON
└── created_at: timestamp

CalendarGroup（日历组）
├── id: string (UUID)
├── name: string
├── icon: string
├── color: string (#HEX)
├── description: string
├── creator_id: string → User.id
├── invite_code: string (唯一邀请码)
├── max_members: int (免费10/付费50)
└── created_at: timestamp

GroupMember（组成员）
├── group_id: string → CalendarGroup.id
├── user_id: string → User.id
├── role: enum (creator/admin/member)
└── joined_at: timestamp

Event（日程）
├── id: string (UUID)
├── title: string
├── description: string
├── start_time: timestamp
├── end_time: timestamp
├── is_all_day: boolean
├── location: string
├── color: string
├── visibility: enum (public/busy/private)
├── repeat_rule: JSON (RRULE格式)
├── lunar_date: string (农历日期，可选)
├── group_id: string → CalendarGroup.id (null=个人日历)
├── creator_id: string → User.id
├── template_id: string → Template.id (可选)
└── created_at: timestamp

Reminder（提醒）
├── id: string (UUID)
├── event_id: string → Event.id
├── user_id: string → User.id
├── remind_at: timestamp
├── status: enum (pending/sent/failed)
└── created_at: timestamp

Template（模板）
├── id: string (UUID)
├── name: string
├── category: string
├── preset_data: JSON
├── is_system: boolean (系统预设 vs 用户自建)
├── creator_id: string → User.id (用户自建时)
└── usage_count: int
```

### 6.2 关系图

```
User ──1:N──> GroupMember ──N:1──> CalendarGroup
User ──1:N──> Event
CalendarGroup ──1:N──> Event
Event ──1:N──> Reminder
Template ──1:N──> Event (可选关联)
```

## 十、风险与应对

| 风险 | 等级 | 应对策略 |
|------|------|---------|
| 微信订阅消息限制导致提醒体验差 | 高 | 设计巧妙的授权引导；提醒用户"添加到桌面"获取系统通知 |
| 冷启动阶段组内人少体验差 | 高 | 强化邀请引导流程；即使1人也能用（个人日历有价值） |
| 腾讯日历强化共享功能 | 中 | 速度取胜+差异化（AI语音、模板、空闲匹配） |
| 语音识别准确率不够 | 中 | 始终有人工确认步骤；提供编辑修改能力 |
| 用户隐私顾虑 | 中 | 可见性三级控制；隐私声明透明化 |
| LLM API成本过高 | 中 | 免费版限次数；优化Prompt减少token消耗 |

## 十一、设计原则

1. **共享优先** — 每个设计决策都问："这如何帮助人们更好地共享日程？"
2. **极简克制** — 不做中华万年历式的功能堆砌，每个功能都要有明确价值
3. **零门槛** — 无需注册（微信登录）、无需教学、无需下载（小程序）
4. **尊重隐私** — 用户完全控制自己的日程可见性，默认私密
5. **本土化** — 不是翻译国际产品，而是为中国用户从零设计
